Ce data challenge propos par SNCF-Transilien a pour objectif de mieux anticiper le nombre de validations quotidiennes dans les gares d'le-de-France. En tant que principal oprateur de trains de banlieue dans cette rgion, SNCF-Transilien fait circuler chaque jour plus de 6 200 trains, transportant des millions de voyageurs. Le nombre de validations de cartes puce augmente rgulirement environ 6% par an entre 2015 et 2019 , ce qui rend crucial de mieux prvoir cette croissance pour adapter les services et amliorer la gestion des flux.

Le but de ce challenge est de construire un modle capable de prdire, pour chaque gare, le nombre de validations journalires sur le moyen et long terme. Il sagit dun problme classique de prvision de sries temporelles, mais la difficult rside ici dans la gestion d'un grand nombre de sries distinctes, correspondant chacune des gares. Pour SNCF-Transilien, l'enjeu est de pouvoir anticiper les besoins futurs en termes de transport et d'ajuster loffre de faon proactive afin de rpondre efficacement la demande croissante.

Nous avons les jeux de donnes :

  • 'train_f_x.csv' : Ce fichier contient les caractristiques (features) relatives aux validations quotidiennes dans 448 gares du rseau SNCF-Transilien, incluant toutes les stations des lignes RER A et RER B. Il couvre une priode allant du 1er janvier 2015 au 31 dcembre 2022, soit un total de 2 922 jours dobservation.

  • 'y_train_sncf.csv' : Ce fichier comprend un index (qui est la combinaison de la date et de l'identifiant de la station) ainsi que la variable cible y, reprsentant le nombre de validations. Cette variable est associe aux caractristiques prsentes dans train_x.csv.

  • 'test_f_x_THurtzP.csv' : Ce fichier contient les mmes caractristiques (features) que train_x.csv, mais pour une priode plus rcente, du 1er janvier 2023 au 30 juin 2023, soit 181 jours. Ce jeu de donnes est destin tre utilis pour prdire les valeurs de y, cest--dire le nombre de validations journalires pour chaque station, en se basant sur le modle entran. \newline

Nous avons les features:

  • date : la date laquelle les validations ont t enregistres, au format AAAA-MM-JJ.

  • station : identifiant anonymis de chaque gare cod par une chane de trois caractres.

  • job : indicateur du Jour Ouvrable de Base (JOB) valant 1 pour les jours de semaine et 0 pour les week-ends.

  • ferie : indicateur pour les jours fris, avec 1 si le jour est fri et 0 sinon.

  • vacances : indicateur pour les vacances scolaires, valant 1 si le jour fait partie des vacances scolaires et 0 sinon.

Ajout de nouvelles features

Pour enrichir les analyses, nous avons dvelopp la fonction create_features_date(), qui dcompose chaque date en plusieurs composantes comme le jour, le mois, lanne, et le jour de la semaine. Cela nous a permis de mieux analyser ces informations de manire spare et de les utiliser pour affiner le modle.

En parallle, nous avons dvelopp la fonction create_features_lags() pour introduire des lags temporels comme nouvelles features. Aprs plusieurs tests (et en tenant en compte les tendances saisonnires), nous avons retenu les lags de 1 jour, 7 jours et 1 mois, car ils ont obtenu les meilleurs rsultats en termes de score et figuraient parmi les plus importants dans les analyses dimportance des features (dtailles plus bas). Ces lags se sont rvls cruciaux pour capturer la nature des donnes temporelles, apportant des informations essentielles pour un entranement optimal du modle.

Fonctions pour crer des features supplmentaires:

In[8]:
def create_features_date(df):
    ''' Create features from date '''
    df['day'] = df['date'].dt.day
    df['month'] = df['date'].dt.month
    df['year'] = df['date'].dt.year
    df['dayofweek'] = df['date'].dt.dayofweek
    return df
In[9]:
def create_features_lags(df_x, df_y):
    ''' Create lag features '''
    target_map = df_y['y'].to_dict()

    df_x['lagDay'] = (
        (df_x.date - pd.DateOffset(days=1))
        .dt.strftime('%Y-%m-%d') + '_' + df_x.station
    ).map(target_map)
    
    df_x['lagWeek'] = (
        (df_x.date - pd.DateOffset(days=7))
        .dt.strftime('%Y-%m-%d') + '_' + df_x.station
    ).map(target_map)

    df_x['lagMonth'] = (
        (df_x.date - pd.DateOffset(months=1))
        .dt.strftime('%Y-%m-%d') + '_' + df_x.station
    ).map(target_map)

    return df_x

estimator = lgb.LGBMRegressor(objective='poisson', random_state=42, max_bin=1000, max_depth=10, verbosity=-1)

param_grid = {
    'n_estimators': [700, 5000, 10000],
    'learning_rate': [0.1, 0.01]
}

grid_search = GridSearchCV(
    estimator=estimator,
    param_grid=param_grid,
    scoring='neg_mean_squared_error',  
    cv= KFold(n_splits=3, shuffle=True),
    verbose=False,
